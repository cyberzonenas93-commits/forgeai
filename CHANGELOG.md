# Changelog

## Unreleased
- **Repair loop learning pass**: The agent now persists repeated-failure memory per task instead of treating each retry as isolated. Validation failures are fingerprinted by class, paths, and exact locations; repeated signatures escalate repair strategy from `targeted_patch` to `widened_context` to `escalated_reasoning`; repair prompts now include recent repair memory so the model is told not to reuse the same broken patch pattern unchanged. Validation also reruns in a narrower failure-aware order first, so repeated test failures prefer test-first reruns while import/type/build failures prefer analyze/build-first reruns before broader suites. Repair quality metrics now track repeated failure events, bottleneck categories, passes to success, and estimated repair cost, and runtime wording now makes the workspace source of truth explicit: task-local workspace or sandbox, never Firestore drafts.
- **Distributed worker plane**: Replaced the last monolithic execution seam with a Firestore-backed worker plane. `runAgentTask` now dispatches `agentWorkerRuns` instead of executing the full agent loop inline, `processDistributedAgentWorker` claims and heartbeats each run, and `recoverStaleAgentWorkers` re-dispatches stalled work by bumping the task `runToken`. Firestore remains the control plane for queueing, locks, approvals, worker leases, metrics, and event streaming, while multiple repos can now execute in parallel through independent worker runs.
- **Cost-aware orchestration**: Added task-level budget snapshots, agent cost ledgers, and profile-aware routing. The runtime now records estimated token and provider-cost activity per stage, tracks remaining task budget, and uses economy/balanced/quality routing profiles to steer planning, context, editing, and repair across `openai`, `anthropic`, and `gemini`.
- **Logical multi-agent coordination**: Added explicit planner/context/editor/validator/repair/git role orchestration on top of the single visible task. The runtime now records logical handoffs and summaries in task metadata so the backend can coordinate specialized phases while the user still experiences one unified coding agent.
- **Broader local validation commands**: Expanded the allowlisted workspace command surface to cover `npm build`, `tsc --noEmit`, and `eslint` when available, in addition to the existing Flutter/Dart/npm validation commands.
- **Repo-native task workspaces**: Replaced the main agent path’s Firestore-backed apply model with task-local cloned repositories. Each agent run now clones the connected repo into an isolated workspace, validates candidate diffs in sandbox copies of that clone, applies approved diffs to the task-local workspace, and performs commit/push/PR follow-ups with real git commands from that workspace. Firestore remains the durable store for task state, approvals, events, execution sessions, and repo metadata, while legacy draft flows are now clearly secondary to the repo-native agent path.
- **Ruthless second pass**: Removed one of the last artificial repo-agent bottlenecks by dropping the extra `10/20` generated-file cap inside repo execution, so the runtime can now use the full editable-wave budget selected by the context planner instead of pretending the repo view is smaller than it is. Repair and guardrail regeneration also no longer stay pinned to the previous execution provider after failures; after the first failed pass the router can promote a stronger provider for recovery work. Repo-map, context-expansion, and diff-generation phases are now recorded as first-class tool executions in task metadata, and the mounted UI now calls out editable waves, live agent runs, and active repo work instead of suggestion/chat language.
- **Pre-approval sandbox validation**: Moved the real validation-and-repair loop ahead of apply approval. Agent runs now materialize sandbox workspaces from the task-local repo clone, apply the candidate diff there, run static/local/remote validation, repair failed drafts before approval, and only ask to write into the task-local workspace after the diff has already passed validation. Added ephemeral workspace support in `functions/src/ephemeral_workspace.ts`, pre-approval validation orchestration in `functions/src/index.ts`, and UI indicators that the diff was validated before apply.
- **Agent hardening second pass**: Tightened the last weak spots that still felt like suggestion mode. Repair passes now promote validation failure paths back into repo exploration, structured repo execution retries now keep going across a wider provider fallback chain instead of stopping after two malformed payloads, and the mounted prompt/editor handoff now frames work as a live repo work item rather than an assistant-style reply.
- **Agent runtime architecture pass**: Added explicit multi-provider routing for the main repo-agent path, expanded `AI_PROVIDER_NAMES` to `openai`, `anthropic`, and `gemini`, created `functions/src/provider_interface.ts` and `functions/src/routing_engine.ts`, and routed repo context planning, execution planning, diff generation, and repair passes through stage-aware provider selection. Added `functions/src/tool_registry.ts` and `functions/src/tool_executor.ts`, recorded tool catalogs and tool-execution history in task metadata, surfaced routed provider/tool information in the Agent UI, and documented the stronger runtime in `AGENT_RUNTIME_ARCHITECTURE.md`.
- **Retry-limit clarity pass**: Tightened the validation/repair loop around the user feeling of “the agent kept trying until it worked or hit a hard limit.” The backend now records explicit retry-limit metadata when repair or guardrail regeneration is exhausted, repair prompts carry recent validation history, and the agent UI now shows repair pass counts, validation-pass counts, and a visible hard-limit state instead of a generic failure.
- **Validation loop second pass**: Hardened the repair loop again so the agent no longer dies immediately on an oversized first diff. Pre-approval guardrail failures now trigger a narrower regeneration pass, repair prompts now include recent validation history instead of only the latest failure string, and manual checks are no longer dispatch-only. Added Firestore-triggered check monitoring in `functions/src/index.ts`, upgraded the Checks dashboard to surface agent-driven versus manual runs plus structured findings, and enriched `ForgeCheckRun` plus repository parsing for validation details.
- **Tool execution and repair loop**: Replaced the old “local consistency plus maybe one remote workflow” validation path with a structured tool suite. Agent runs now execute workspace consistency checks, static repo validation over the task-local cloned draft, and GitHub CI validation on a fresh validation branch containing the actual edited draft. Validation emits per-tool lifecycle events, persists structured findings/history on the task, and feeds those findings back into iterative repair passes. Added `functions/src/agent_validation_tools.ts`, upgraded the validation runner in `functions/src/index.ts`, and expanded the Agent task details/timeline UI to show tool results, findings, validation attempts, and validation branch metadata.
- **Whole-repo awareness second pass**: Pushed the repo-context system further away from shallow subset behavior by raising GitHub tree sync breadth to `10000` entries, adding explicit sync-coverage notices when the remote tree is still truncated, expanding deep mode into module/dependency-neighborhood exploration instead of only larger file budgets, widening execution-memory retention for explored paths/modules/passes, and feeding architecture zones, full module indexes, and focused-module details into planner and execution prompts.
- **Whole-repo awareness architecture**: Replaced the old bounded context path with a size-aware repo knowledge map, module and architecture summaries, import/reverse-import graph traversal, persistent repo context snapshots, progressive context-expansion passes, and durable run-level execution memory that survives approvals and repair loops. Live agent tasks now stream exploration-pass metadata such as repo size class, context strategy, focused modules, architecture findings, and execution-memory summaries while the run is still inspecting the repository.
- **User-perspective agent pass**: The Agent tab now keeps the user-selected queued run in focus instead of snapping back to the currently active run, shows a dedicated queued-behind-active banner, and makes it clear when a newly submitted prompt is planned, waiting, and still visible while another task owns the workspace lock.
- **Visible repo understanding**: The live console now exposes a dedicated repo-understanding panel with mapped-file counts, editable focus files, dependency context, global repo anchors, and whole-repo-inline versus expanded-context strategy so the run feels like deep repo work instead of a hidden retrieval step.
- **Live runtime state surfaced earlier**: Repo inspection now writes selected-file, inspected-file, repo-size, and context-strategy metadata onto the task record during execution rather than only after diff generation. The “Now” event card and queue cards also surface richer plan/context metadata so the console feels active before approval time.
- **Second parity pass**: Removed the dead `AskScreen` / prompt-intent UI, collapsed the mounted review flow around repo execution sessions only, and cut the last live single-file change-request branch out of the Flutter app so the visible AI experience is now one agent-first path.
- **Explicit run planning**: Agent tasks now write an execution plan before repo inspection begins. The runtime stores AI-generated follow-up intent plus ordered run steps in task metadata, surfaces that plan in the live timeline/details UI, and uses the refined follow-up plan for commit / PR / merge / deploy continuity instead of relying only on keyword inference.
- **Agent console clarity**: Review and task screens now expose inspected-file breadth, global-context files, planning summaries, and planned follow-up actions. Queue cards, active headers, and file-activity rows no longer clip important task/file text in the live agent console.
- **Agent parity upgrade**: Increased GitHub repo sync breadth from a shallow `200`-file tree cap to batched larger-tree ingestion, fixed summary extraction so import/symbol signals use raw content, and strengthened repo execution with repo manifests, whole-repo-inline eligibility checks, global context files, a context-expansion planner, and a second execution-planning pass before code generation.
- **Iterative validation loop**: Upgraded the durable agent runtime from one-shot diff generation into `apply -> validate -> repair -> repeat` behavior. The backend now validates sandbox and task-local workspace drafts, can dispatch and poll GitHub validation workflows, ingests failure context, and automatically generates or applies repair passes until success or retry exhaustion.
- **Unified agent entry**: Prompt/Ask submissions now queue durable agent tasks instead of bypassing the live runtime with a direct one-shot repo execution call. That brings queueing, live events, approvals, and continuation behavior to the prompt surface as well.
- **Execution session visibility**: Repo execution sessions now persist and expose broader context metadata including `inspectedFiles`, `globalContextFiles`, `repoOverview`, `wholeRepoEligible`, and `planningSummary`, so the client can represent what the agent actually searched instead of only what it edited.
- **True agent mode**: Replaced the prompt-and-reply execution path with persisted Firestore-backed agent tasks, workspace locks, queued prompts, live event streaming, multi-step backend execution, approval checkpoints, cancellation/pause controls, and long-running task recovery. Added `enqueueAgentTask`, `cancelAgentTask`, `pauseAgentTask`, `resolveAgentTaskApproval`, Firestore trigger `runAgentTask`, new Agent tab UI, task/event/approval models, workspace-controller subscriptions, and architecture docs (`AGENT_MODE_ARCHITECTURE.md`, `TASK_QUEUE.md`, `LIVE_EXECUTION_UI.md`).
- **Live execution UX**: Rebuilt the Agent tab into a mobile-first work session console with a pinned prompt composer, animated active-step header, files-touched panel, structured timeline console, queue preview + queue sheet, task details screen, success/failure states, and approval checkpoint modal/actions so the product feels like an active coding agent instead of chat.
- **Live execution UX polish**: Tightened hierarchy, spacing, labels, and state treatment across the work session. Active runs are more visually obvious, queued tasks are clearer about ordering/next-up status, approval checkpoints read like intentional gated actions, and completion/failure cards now feel more premium and developer-focused.
- **Live execution UX audit**: Removed the misleading fallback that could make historical runs look active, pulled the run queue higher in the hierarchy, clarified workspace-busy queue messaging, and renamed remaining prompt-like UI labels so the experience reads as a live agent console instead of chat.
- **App Store review**: Documented demo credentials (`test@codecatalystai.com`) in `APPLE_REVIEW_NOTES.md` and `docs/APP_STORE_REVIEWER_ACCOUNT.md`. Backend allowlists that email for unlimited wallet usage and adds scheduled `purgeExpiredAppStoreReviewerTestAccounts` to delete the Auth user 30 days after creation (Firestore cleanup via existing `deleteUserDataOnAuthDelete`).
- **New project (AI)**: Repositories tab can create a new GitHub or GitHub repository, generate a starter file set with OpenAI, commit files remotely, sync into the workspace, and select the new repo. Callable `createProjectRepository`, wallet action `ai_project_scaffold`, Flutter `NewAiProjectScreen` + `ForgeWorkspaceController.createProjectWithAi`.

Billing abstraction (`ForgeBillingService`), mock implementation, paywall and token pack screens, wallet upgrade/get-tokens CTAs. Revenue/cost telemetry via existing opsMetrics and `forge_paywall_viewed` / `forge_token_packs_viewed` analytics. Docs: TOKEN_ECONOMICS.md, PRICING_MODEL.md, BILLING_ARCHITECTURE.md, APPLE_IAP_PLAN.md, REVENUE_SIMULATION.md.
